# QQ 机器人进阶功能技术架构与实现方案

本文档作为 PRD 的技术补充，详细说明 2.4 节所列进阶功能的具体实现路径与技术选型。

## 1. 联网搜索功能

**实现目标**：解决大模型训练知识库过时的问题，使其能回答最新资讯。

**技术实现途径**：
*   **方案一（推荐）：使用内置搜索的 LLM API**。
    *   例如：直接调用支持搜索的 Qwen-Max 或 Moonshot，这些模型在 API 侧已内置了联网搜索能力，开发者只需在请求参数中开启对应 flag（如 `enable_search: true`），开发成本极低。
*   **方案二：Agentic/Tool Use (函数调用)**。
    *   引入 DuckDuckGo API (如 `duckduckgo-search` 库) 或 Bing Search API。
    *   利用模型的 Function Calling 能力：当需要搜索时，模型返回工具调用请求 -> 机器人执行搜索 -> 将搜索结果作为上下文返回给模型 -> 模型生成最终回答。
    *   **优点**：过程完全由开发者把控，可自定义搜索源。

## 2. 群聊消息总结

**实现目标**：提取群内近期聊天记录并生成摘要，帮助成员快速了解上下文。

**技术实现途径**：
*   **消息拦截与临时存储**：
    *   利用 NoneBot2 的全局消息钩子 (Message Event Hook)，将群聊消息（至少包含发言人、时间和文本内容）存入内存队列（如 `collections.deque`，针对每个群维护固定长度，例如最近 200 条）或轻量级数据库。
*   **指令触发与拼装**：
    *   用户发送 `/总结` 时，机器人从该群的队列中取出最近的 N 条记录。
    *   将这些记录格式化成统一形式（例如：`张三(10:00): 吃饭了吗\n李四(10:01): 吃了`），拼接成一个长文本 Prompt。
*   **LLM 提炼**：
    *   设定 System Prompt，要求模型对上述长文本进行归纳，提取核心话题和结论，并通过 LLM API 获取摘要后发送回群聊。

## 3. 图生图与图像处理 (ComfyUI 扩展)

**实现目标**：支持对用户发送的图片进行再创作（如转二次元、上色、超分）。

**技术实现途径**：
*   **图片获取**：
    *   当机器人收到带图片的指令（如 `/图生图 [描述] [图片]`）时，从 QQ 消息中解析出图片的 URL，并使用 `httpx` 下载该图片到本地临时目录。
*   **上传到 ComfyUI**：
    *   调用 ComfyUI 的 `/upload/image` API，将图片上传至 ComfyUI 服务端，获取服务端的文件名参考。
*   **替换 Workflow 参数**：
    *   准备专门的图生图/超分 API JSON 工作流。
    *   在工作流的 JSON 结构中定位到 LoadImage 节点（通常对应 `LoadImage` 类节点），将其 `image` 字段对应的值替换为上一步得到的文件名。
*   **生成与返回**：
    *   后续流程同文生图：发送 Prompt，通过 WebSocket 监听进度，完成后下载并回传。

## 4. 长期记忆与用户画像

**实现目标**：超越单次会话的上下文，针对特定用户形成长期记忆。

**技术实现途径**：
*   **数据库选型**：
    *   推荐使用 **SQLite** 搭配 ORM 框架（如 SQLAlchemy 或 Tortoise ORM）。数据量小，无需额外部署数据库服务。
*   **数据结构设计**：
    *   维护一张 `UserArchive` 表，包含列：`user_id` (QQ号), `nickname`, `preferences` (JSON 格式，存喜好等), `summary_memory` (对该用户的长期印象总结段落)。
*   **记忆更新与提取机制**：
    *   **提取**：在每次发生对话前，查询该用户的 `UserArchive`，若存在 `summary_memory`，则将其作为系统 Prompt 的一部分（例如：“该用户叫张三，喜欢赛博朋克风格...”）一并传给大模型。
    *   **更新**：每隔一定周期（比如每 50 轮对话，或每天结束），启动后台异步任务，将最近的对话记录送给大模型进行“信息提纯”，让它总结出新发现的用户偏好，并更新写入数据库中。

## 5. 定时与推送服务

**实现目标**：实现定时发送“早报新闻”、“每日摸鱼图”等功能。

**技术实现途径**：
*   **调度框架选型**：
    *   如果使用 NoneBot2 框架，推荐直接集成 `nonebot_plugin_apscheduler`。它底层是对 `APScheduler` 的封装，天然支持 Cron 表达式和异步任务。
*   **任务逻辑**：
    *   **新闻早报**：定时任务触发 -> 调用第三方免费新闻接口 (如 `果创云`、`聚合数据` 或使用爬虫抓取每日必看) -> 构建图文消息 -> 调取 NoneBot 的 `bot.send_group_msg` 广播到配置的群组。
    *   **每日摸鱼图**：定时任务触发 -> 本地或远程发起一个 ComfyUI 的固定 prompt 随机种子生图请求 -> 生成完毕后推送到群聊。
*   **群组配置管理**：
    *   在数据库或配置文件中维护一张“已订阅群组”列表，提供 `/订阅早报`、`/退订早报` 的管理指令供群馆员操作。

## 6. 大学生课表解析与上课提醒

**实现目标**：解析用户上传的课表。在用户当前处于上课时间段并且触发聊天时，将上课状态作为上下文传递给大语言模型，由其生成符合人设的自然劝学提醒，然后再回复正常聊天。

**技术实现途径**：
*   **课表获取与解析 (WakeUp 分享口令接入)**：
    *   **指令输入**：如用户发送 `/导入课表 [WakeUp分享口令文本]`（例如解析出 32 位 Hex 口令代码 `493f666c20f54f4f9dd0f19ad614f228`）。
    *   **接口抓取**：基于提取出的分享口令，后台使用 `httpx` 调用 WakeUp 课程表的云端 API 接口。接口通常会直接返回结构化的 JSON 数据（包含开学日期、每天的作息时间表、具体的课表信息等）。彻底抛弃繁重的 OCR 识别与本地文件解析逻辑。
*   **持久化数据存储**：
    *   **数据库设计**：在现有的 SQLite 中创建表（例如 `UserSchedules`）。存储 `user_id`、课程开始周次与结束周次、单双周标记、`day_of_week` (星期)、作息开始和结束时间 (如 `08:00` - `08:45`)、`course_name` 等。
    *   记录该用户的“**开学日期**”（第一周第一天），这是精确推算单双周与当前上课状态的核心原点。
*   **全局状态拦截与指令拦截**：
    *   **事件预处理器 (EventPreProcessor)**：不局限于普通聊天指令。通过在 NoneBot 的全局消息、乃至指令钩子阶段，拦截所有触发行为（不管用户是在聊天、生图还是提问）。
    *   **动态推演逻辑**：获取发信时的系统精确时间。结合数据库获取用户的“开学日期”，推算当前是第几周、星期几。随后比对当日作息时间，判断此刻是否落入某节课内，如果是，则抛出当前上课的 `course_name`。
*   **大语言模型应对与复合组装**：
    *   **聊天场景**：如果判定用户在上课，立刻在请求构建中向 **System Prompt** 临时注入提示：“该用户目前正在上《{ course_name }》，请你先非常自然、符合人设地提醒他收心听课，再顺着他的话接着往下聊。”
    *   **非聊天/工具指令场景（如 /画图）**：由于工具指令（如排队生图）本身不调大模型，此时若处于上课状态，机器人除执行工具本身逻辑外，会额外抛出一个**极轻量级的独立 LLM 请求**，令模型针对当前课程单独生成一句极短的劝学句（如：“上着《马克思》还喊我画图？”）。随后将这句话前置拼接到该工具的回应文本前返回给用户，实现全场景无死角提醒。
