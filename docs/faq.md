# FAQ

这里整理了这个项目最常见的使用问题，优先覆盖真实使用里最容易卡住的点。

## 1. 启动时报 `error while attempting to bind on address ('0.0.0.0', 8080)`

说明端口被占用了。

排查方式：

```powershell
netstat -ano | findstr :8080
taskkill /PID <PID> /F
```

或者直接换一个 `PORT`，同时把 NapCat / Lagrange 的反向 WebSocket 地址一起改掉。

## 2. 机器人没有回复

先按这个顺序检查：

1. NapCat / Lagrange 是否已经连接到正确的反向 WS
2. `.env` 里的 `PORT` 是否和 OneBot 配置一致
3. 当前群是否被加入 `GROUP_BLACKLIST`
4. 机器人是否真的被艾特，或者命令前缀是否正确
5. 是否同时启动了多个 Bot 进程

## 3. 机器人一次回两条消息

优先检查是否开了多个进程，或者同时连了多个适配器实例。

常见原因：

- 你在两个终端都运行了 `python bot.py`
- OneBot 客户端重复连接到同一个 Bot
- 历史旧进程没有关闭干净

## 4. `/图生图` 提示没有带图

图生图支持两种用法：

1. 指令消息本身带图，再写 `/图生图 提示词`
2. 回复某张图片消息，再发送 `/图生图 提示词`

如果还不行，优先检查：

- `COMFYUI_IMG2IMG_WORKFLOW` 路径是否正确
- 图生图工作流里是否保留了 `LoadImage` 相关节点
- 当前图片尺寸是否超过项目限制

## 5. 文生图 / 图生图结果风格不对

这通常不是代码坏了，而是工作流和模型本身决定了输出风格。

建议检查：

- 当前选择的模型和 LoRA 是什么
- 是否还启用了之前的 LoRA
- 工作流里是否写死了额外提示词
- 是否保留了不想要的放大或二次处理节点

## 6. 模型列表或 LoRA 列表太少

项目支持读取本地可见的模型和 LoRA，但前提是：

- ComfyUI 运行正常
- 代码能访问到 ComfyUI 当前扫描到的模型目录
- `DRAW_LIST_ALL_LOCAL_STYLES=true`

如果你刚添加了新模型，先重启 ComfyUI，再重启机器人。

## 7. `/搜索` 效果差，搜不出有效信息

先看你现在实际使用的是哪一路搜索能力。

改进建议：

- 给 `SERPER_API_KEY` 配真实值
- 检查外网访问是否正常
- 对新闻类内容优先用更明确的关键词
- 确认请求没有被目标站点限流

## 8. `/导入课表` 成功了，但课表不完整或不准确

当前推荐优先使用 WakeUp 32 位口令导入。

如果你导入的是 PDF、图片或文本，解析质量会明显依赖原始格式。

优先检查：

- 是否真的是 WakeUp 分享口令，而不是普通文本
- 学期开始日期是否正确
- 原 PDF 是否为可复制文本，而不是纯截图型 PDF
- 课程是否包含单双周、节次合并、跨周等复杂格式

如果 PDF 本身排版混乱，代码再努力也只能做近似解析，这时最好直接换 WakeUp 口令。

## 9. 识图功能没触发

当前识图触发条件通常是：

- 艾特机器人且消息里带图
- 或者回复一张图并发送识图相关内容

如果没触发，检查：

- 图片是否真的被 OneBot 上报
- 当前消息事件里有没有拿到图片 URL
- 你配置的模型是否支持视觉输入

## 10. 角色扮演突然不自然，或者命令回复像“机器回执”

优先检查这几项：

- `LLM_SYSTEM_PROMPT`
- `COMMAND_ROLEPLAY_ENABLED`
- `COMMAND_ROLEPLAY_PREFIX`
- `COMMAND_ROLEPLAY_SUFFIX`

如果你替换过整段人设，最好把聊天、识图、搜索、课表导入各测一遍，因为这些功能都会共享角色表达层。

## 11. 改了 `.env` 没效果

最常见原因就是机器人没重启。

推荐流程：

```powershell
.\scripts\check.ps1
python bot.py
```

如果还是不对，再确认你改的是项目根目录的 `.env`，不是别的副本。
